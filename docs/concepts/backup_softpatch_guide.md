# 백업 데이터 복원 및 소프트패칭 가이드북 (Backup Recovery & Soft-Patching Guide)

이 문서는 HASTE Company의 클라우드 데이터베이스(Supabase) 및 로컬 시뮬레이터 환경에서 시스템을 기동, 복원, 동기화할 때 원본 운영 데이터를 보존하고 충돌을 예방하기 위한 개발 무결성 매뉴얼입니다.

---

## 📌 1. 소프트 패칭(Soft-Patching) 무조건 준수 원칙

상용 클라우드 데이터베이스를 업데이트하거나 시스템을 복원할 때, 기존 운영 데이터를 비우고 덮어쓰는 하드 리셋(Hard Reset / Truncate) 방식은 전면 금지하며 오직 **'소프트 패칭(Soft-Patching)'**을 절대 기준으로 적용합니다.

*   **동작 방식**: 
    1.  서버 초기 실행 및 데이터 시딩 시, 비어 있는 테이블에만 기본 초기 데이터를 삽입(`INSERT`)합니다.
    2.  이미 기존 데이터가 존재하는 경우, 절대 기존 행(`Row`)을 지우지 않고 신규 데이터 항목만 점진적으로 추가하거나 변경 사항만 안전하게 업데이트(`UPDATE`)합니다.
*   **관련 코드**: [database.ts](file:///c:/Users/김성규/Desktop/HASTE-Company/cafehaste-web/server/database.ts) 내의 `syncDraftsBackup`, `register-song` 등에서 DB 덮어쓰기 오작동을 차단하는 보호 로직을 강제 적용합니다.

## 📌 2. 로컬 시뮬레이션 환경 격리 규칙 (Local Isolation)

개발 도중 실수로 실서버 데이터를 변조하거나 상용 DB가 오염되는 것을 차단하기 위해, 로컬 개발 환경(`localhost`)과 운영 클라우드 환경을 완벽하게 격리합니다.

*   **SELECT (읽기) 허용**: 로컬 프로그램 및 웹 서버는 실서버 데이터베이스에서 데이터를 정상적으로 읽어와 화면을 렌더링할 수 있습니다.
*   **CUD (쓰기/수정/삭제) 차단 및 우회**:
    *   로컬 개발 환경에서 발생하는 모든 생성(`Create`), 수정(`Update`), 삭제(`Delete`) 동작은 실제 Supabase 클라우드 데이터베이스로 직접 쿼리하지 않습니다.
    *   해당 쓰기 요청들은 로컬 프로젝트 내의 **백업 JSON 파일 (`local_*.json`)**로 강제 우회시켜 디스크 파일로만 갱신되도록 쿼리 핸들러를 가동합니다.
*   **관련 파일**: `local_members.json`, `local_licenses.json`, `local_menu_items.json` 등 10여 개의 로컬 백업 파일에만 쓰기를 적용하여 안전망을 확보합니다.

---

## 📌 3. `local_*.json` 백업 동기화 및 복구 절차

로컬에서 시뮬레이션한 가맹점 정보 및 콘텐츠 설정을 배포하거나, 클라우드 DB가 마비되어 로컬 백업에서 데이터를 복구해야 할 때 다음 절차를 준수합니다.

1.  **동기화 검증**:
    *   로컬 JSON 파일을 클라우드 DB에 부어넣기 전에, 반드시 중복 검사 및 ID 맵핑 검사를 수행합니다.
    *   특히 가상 점주 `member_id`가 `90000`대 규격을 어기고 일반 회원 테이블과 겹치지 않는지 선제적으로 점검합니다.
2.  **클라우드 푸시 프로세스**:
    *   복구 스크립트 실행 시, 클라우드 DB의 `web_membership_users` 및 `web_store_licenses` 상태가 가입자 가이드북에 적힌 **[인증 완료]** 상태와 일치하도록 보장해야 합니다.
3.  **캐시 무효화 (Invalidation)**:
    *   데이터 동기화 및 갱신이 최종 완료된 경우, 클라우드 및 로컬의 인메모리 싱글톤 캐시(`storeVerifyCache`)를 즉시 소거(`delete`)하여 가맹점 키오스크가 최신 상태의 승인 데이터를 즉각 조회하도록 처리합니다.
